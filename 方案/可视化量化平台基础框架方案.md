# 可视化量化平台基础框架方案

## 1. 建设目标

当前第一阶段目标是搭建一个可扩展的量化研究与回测平台，先支持基于历史股票数据的策略算法回算，后续再逐步扩展到可视化分析、参数优化、模拟交易、实盘交易和风控监控。

平台的核心能力可以分为五层：

1. **数据层**：获取、清洗、存储和查询历史行情、财务、指数、因子等数据。
2. **策略层**：定义策略逻辑，包括择时、选股、调仓、止盈止损、仓位管理等。
3. **回测层**：按时间推进，模拟交易、撮合、资金变化、持仓变化和交易成本。
4. **分析层**：计算收益、回撤、夏普、胜率、换手率、风险暴露等指标。
5. **可视化层**：展示净值曲线、持仓变化、交易点位、收益分布和策略对比。

第一阶段建议优先完成“数据接入 + 单策略回测 + 基础绩效指标 + 简单图表展示”，不要一开始就做过重的实盘和复杂因子平台。

---

## 2. 推荐技术架构

如果你主要使用 Python，可以采用如下基础技术栈：

| 模块 | 推荐方案 | 说明 |
|---|---|---|
| 编程语言 | Python | 量化生态成熟，数据处理方便 |
| 数据处理 | pandas / numpy | K 线、收益率、指标计算 |
| 数据库存储 | SQLite 起步，后续 PostgreSQL | SQLite 适合本地原型，后续可迁移 |
| 行情数据源 | AkShare / Tushare / BaoStock / 自有 CSV | A 股历史数据常用来源 |
| 回测引擎 | 自研轻量引擎起步 | 更容易理解交易过程和定制规则 |
| 可视化 | matplotlib / plotly / echarts | plotly 和 echarts 更适合交互图表 |
| Web 后端 | FastAPI | 后续做可视化平台接口 |
| Web 前端 | Vue / React | 展示策略配置、回测结果和图表 |
| 任务调度 | APScheduler / Celery | 后续用于数据更新和批量回测 |

第一阶段可以先不做完整前后端，建议先用 Python 脚本或 Notebook 把回测核心跑通，再把能力封装成 API 和页面。

## 3. 第一阶段核心模块设计

### 3.1 数据模块

数据模块负责统一管理历史股票数据，避免策略直接依赖具体数据源。

#### 主要职责

- 拉取股票历史日线数据。
- 清洗字段名和数据类型。
- 处理停牌、缺失值、复权价格等问题。
- 保存到本地数据库或文件。
- 提供统一查询接口。

#### 推荐数据字段

日线行情表 `stock_daily`：

| 字段 | 说明 |
|---|---|
| symbol | 股票代码，例如 000001.SZ |
| trade_date | 交易日期 |
| open | 开盘价 |
| high | 最高价 |
| low | 最低价 |
| close | 收盘价 |
| pre_close | 前收盘价 |
| volume | 成交量 |
| amount | 成交额 |
| adj_factor | 复权因子，可选 |

股票基础信息表 `stock_basic`：

| 字段 | 说明 |
|---|---|
| symbol | 股票代码 |
| name | 股票名称 |
| exchange | 交易所 |
| industry | 行业 |
| list_date | 上市日期 |
| delist_date | 退市日期，可选 |

交易日历表 `trade_calendar`：

| 字段 | 说明 |
|---|---|
| trade_date | 交易日期 |
| is_open | 是否开市 |

#### 数据层接口示例

```python
class DataPortal:
    def get_price(self, symbol, start_date, end_date, fields=None):
        pass

    def get_symbols(self, date=None):
        pass

    def get_trade_calendar(self, start_date, end_date):
        pass
```

策略和回测引擎只调用 `DataPortal`，不要直接读取 CSV 或调用第三方数据 API。

### 3.2 策略模块

策略模块负责生成交易信号。建议先设计一个基础策略接口，所有策略都按同一种形式接入回测引擎。

#### 策略基础接口

```python
class Strategy:
    def init(self, context):
        pass

    def on_bar(self, context, data):
        pass
```

其中：

- `init`：初始化策略参数、指标窗口、股票池等。
- `on_bar`：每个交易日调用一次，根据当天数据决定买卖。
- `context`：保存账户、持仓、当前日期、参数等信息。
- `data`：当前交易日可见的行情数据。

#### 策略返回动作示例

```python
context.order_target_percent("000001.SZ", 0.5)
context.order_target_percent("000002.SZ", 0.5)
```

表示把资金 50% 配置到平安银行，50% 配置到万科 A。

#### 第一阶段建议实现的样例策略

1. **均线择时策略**
   - 当短期均线上穿长期均线时买入。
   - 当短期均线下穿长期均线时卖出。

2. **动量选股策略**
   - 每月选择过去 N 日涨幅最高的前 K 只股票。
   - 等权买入，定期调仓。

3. **低波动策略**
   - 每月选择过去 N 日波动率最低的前 K 只股票。
   - 等权买入，定期调仓。

4. **简单多因子策略**
   - 因子包括动量、波动率、成交额、市值等。
   - 对因子标准化打分，选择综合分最高的股票。

### 3.3 回测引擎模块

回测引擎是第一阶段最重要的核心模块。它负责按照时间顺序模拟策略执行。

#### 回测流程

1. 加载回测参数：起止日期、初始资金、股票池、手续费、滑点等。
2. 加载交易日历。
3. 初始化账户、持仓和策略对象。
4. 按交易日循环：
   - 获取当前日期可用数据。
   - 调用策略 `on_bar` 生成订单。
   - 订单撮合成交。
   - 更新现金、持仓、市值和总资产。
   - 记录当日账户快照。
5. 回测结束后生成绩效报告。

#### 回测配置示例

```python
backtest_config = {
    "start_date": "2020-01-01",
    "end_date": "2024-12-31",
    "initial_cash": 1000000,
    "benchmark": "000300.SH",
    "commission_rate": 0.0003,
    "slippage_rate": 0.0005,
    "frequency": "daily"
}
```

#### 账户对象

```python
class Portfolio:
    cash: float
    positions: dict
    total_value: float
    market_value: float
```

#### 持仓对象

```python
class Position:
    symbol: str
    quantity: int
    avg_cost: float
    market_value: float
    unrealized_pnl: float
```

#### 订单对象

```python
class Order:
    symbol: str
    side: str  # buy / sell
    quantity: int
    price: float
    status: str
```

#### 成交对象

```python
class Trade:
    symbol: str
    side: str
    quantity: int
    price: float
    commission: float
    trade_date: str
```

#### 撮合规则建议

第一阶段可以简化为：

- 策略在 T 日收盘后生成信号。
- 订单在 T+1 日开盘价成交，或用 T+1 日 VWAP / 收盘价近似。
- 买入时检查现金是否足够。
- 卖出时检查持仓是否足够。
- A 股按 100 股整数手交易。
- 手续费按成交金额比例计算。
- 卖出可额外计算印花税。
- 暂停交易、涨跌停无法成交的逻辑可以第二阶段再增强。

### 3.4 绩效分析模块

回测完成后，需要把每日净值和交易记录转成可解释的绩效指标。

#### 基础指标

| 指标 | 说明 |
|---|---|
| 累计收益率 | 策略总收益 |
| 年化收益率 | 折算到年度的收益 |
| 最大回撤 | 历史最高净值到最低净值的最大跌幅 |
| 夏普比率 | 单位波动承受下的超额收益 |
| 波动率 | 收益率标准差年化 |
| 胜率 | 盈利交易占比 |
| 盈亏比 | 平均盈利 / 平均亏损 |
| 换手率 | 调仓频率和交易活跃度 |
| Alpha | 相对基准的超额收益 |
| Beta | 对基准波动的敏感度 |

#### 指标计算输入

建议回测引擎至少输出三张结果表：

1. `equity_curve`：每日资产曲线。
2. `positions_history`：每日持仓快照。
3. `trades`：每笔成交记录。

`equity_curve` 示例：

| trade_date | total_value | cash | market_value | daily_return | benchmark_return |
|---|---:|---:|---:|---:|---:|
| 2024-01-02 | 1000000 | 1000000 | 0 | 0 | 0.001 |
| 2024-01-03 | 1004500 | 200000 | 804500 | 0.0045 | 0.002 |

## 6. 策略开发规范

为了后续可扩展，建议从一开始就给策略设定统一规范。

### 策略应包含

```python
class MyStrategy(Strategy):
    name = "my_strategy"

    params = {
        "short_window": 5,
        "long_window": 20,
        "position_ratio": 1.0
    }

    def init(self, context):
        pass

    def on_bar(self, context, data):
        pass
```

### 策略不应直接做的事

- 不直接读写数据库。
- 不直接修改账户现金和持仓。
- 不直接计算最终绩效报告。
- 不直接依赖某个特定数据源。

策略只负责“根据当前可见数据发出交易意图”，账户变化交给回测引擎处理。

## 7. 避免回测失真的关键点

量化回测最容易出现的问题是“看起来很赚钱，实际不可交易”。第一阶段就要注意以下问题。

### 7.1 未来函数

策略在某一天只能使用当时已经知道的数据，不能使用未来数据。

常见错误：

- 用当天收盘价生成信号，又假设当天收盘价成交。
- 用完整样本计算因子排名，再回到历史日期交易。
- 财务数据没有考虑发布日期。

建议：

- T 日收盘后生成信号，T+1 日成交。
- 所有滚动指标都只使用当前日期之前的数据。
- 财务数据后续要加入公告日期。

### 7.2 幸存者偏差

不能只使用当前仍然上市的股票池，否则会忽略退市股票，导致历史收益被高估。

第一阶段如果数据源暂时不好处理退市，可以先在报告中标记“未处理幸存者偏差”。

### 7.3 复权处理

做技术指标和收益率计算时，一般使用前复权价格更方便。做真实成交模拟时，要注意价格、股数、分红送股之间的一致性。

第一阶段建议统一使用前复权价格计算信号和净值。

### 7.4 交易成本

必须纳入：

- 手续费。
- 滑点。
- 印花税。
- 最小交易单位。

否则高频调仓策略会被明显高估。

### 7.5 流动性约束

后续需要限制单日成交金额占股票成交额的比例，例如不超过当日成交额的 5%。第一阶段可以先记录换手率和成交金额，为后续扩展做准备。

## 8. 后端 API 设计草案

## 12. 第一版可以采用的简化假设

为了尽快跑通，不建议一开始就过度真实。可以先采用这些假设：

- 使用日线数据。
- 使用前复权价格。
- T 日生成信号，T+1 日开盘成交。
- 不考虑涨跌停和停牌，后续再补。
- 不考虑分红现金流，先用复权价格近似。
- 股票交易按 100 股整数手。
- 手续费、滑点、印花税使用固定比例。
- 账户不允许融资，现金不足则按可买数量成交。

这些假设必须在回测报告中展示，避免误解结果。

## 13. 一个最小回测伪代码

```python
class BacktestEngine:
    def __init__(self, data_portal, strategy, config):
        self.data_portal = data_portal
        self.strategy = strategy
        self.config = config
        self.portfolio = Portfolio(config["initial_cash"])
        self.trades = []
        self.equity_curve = []

    def run(self):
        context = Context(self.portfolio, self.config)
        self.strategy.init(context)

        calendar = self.data_portal.get_trade_calendar(
            self.config["start_date"],
            self.config["end_date"]
        )

        for trade_date in calendar:
            context.current_date = trade_date
            data = self.data_portal.get_current_data(trade_date)

            self.strategy.on_bar(context, data)
            orders = context.get_orders()

            trades = self.match_orders(orders, data)
            self.portfolio.update(trades, data)

            self.trades.extend(trades)
            self.equity_curve.append(self.portfolio.snapshot(trade_date))

        return BacktestResult(
            equity_curve=self.equity_curve,
            trades=self.trades,
            metrics=calculate_metrics(self.equity_curve)
        )
```

## 14. 后续扩展方向

当第一阶段稳定后，可以继续扩展：

1. **因子研究平台**
   - 因子 IC、Rank IC、分层回测、因子相关性。

2. **参数优化**
   - 网格搜索、贝叶斯优化、 walk-forward 分析。

3. **组合优化**
   - 均值方差、风险平价、行业中性、市值中性。

4. **风控系统**
   - 最大仓位限制、行业暴露限制、止损、止盈、回撤控制。

5. **模拟交易**
   - 每日自动生成交易建议。
   - 与真实行情同步更新。

6. **实盘接口**
   - 接券商柜台或交易 API。
   - 实盘前必须完成权限、风控、日志、异常处理和人工确认机制。

7. **任务系统**
   - 定时更新数据。
   - 批量运行策略。
   - 保存历史回测结果。

## 15. 总结建议

第一阶段的重点不是做出一个完整漂亮的平台，而是把“历史数据 -> 策略信号 -> 模拟成交 -> 账户净值 -> 绩效分析”这条链路跑通。

建议先做一个轻量自研回测核心，原因是：

- 能清楚掌握回测假设。
- 后续容易接入自己的可视化平台。
- 策略接口、账户系统和结果格式都能按自己的需求设计。

最小可行目标：

1. 用本地历史日线数据跑一个均线策略。
2. 输出交易记录和每日净值。
3. 计算基础绩效指标。
4. 生成一份 HTML 或图表报告。

完成这个闭环后，再逐步做多股票、多策略、Web API 和前端可视化。
